# Plan: Unified Outlined Border Style Across Components

**Date:** 2026-04-27  
**Status:** Proposal

---

## Problem

When composing bordered components side by side — for example a `Select` next to an `outlined` `Button` — the borders do not match. The root cause is a systematic naming asymmetry:

- `Button` with `variant="outlined" themeColor="primary"` draws its border color from the component-variant-specific token `borderColor-Button-primary-outlined`, which resolves to `$color-primary-600` in the default theme.
- `Select` (and every other form input) draws its default border color from `borderColor-Input` → `borderColor-Input-default` → `$colorSurface200` (a light grey), with no concept of "outlined" as a variant.
- The `borderColor` inline prop on `Select` accepts a raw CSS value and overrides the component's SCSS-managed border. It is completely disconnected from the theme token system.

There is no single token that means "the active/normal border color of a primary-outlined component" that can be referenced by both a `Button` and a `Select`.

**Concrete example:**

```xml
<Select id="mnuCustomerFilter" borderColor="blue" />
<Button icon="plus" label="New Customer" variant="outlined" />
```

The user has to hard-code `"blue"` and guess whether it matches the Button's `$color-primary-600`. Any theme change breaks the match.

---

## Goal

1. Introduce a **shared semantic token** `borderColor-outlined` (or `borderColor-interactive-outlined`) that represents "the border color an element should have in an outlined/default style". Form inputs that render a permanent visible border — `Select`, `TextBox`/`Input`, `AutoComplete`, `MultiSelect`, `NumberBox`, `DatePicker`, etc. — default to this token.
2. `Button variant="outlined"` already produces a per-themeColor token (`borderColor-Button-primary-outlined`). Ensure that token **references** the semantic token as its default, so both components read from the same resolved color without custom theming.
3. Where appropriate, add a `variant` prop to `Select` (and to other primary form inputs) mirroring the `Button` variant vocabulary (`outlined` / `ghost` / …), so users can write `variant="outlined"` on any input and get the same visual affordance as an outlined Button.
4. Keep all changes backward-compatible. No existing markup breaks.

---

## Background

### Current token structure

**Button outlined border:**

```
borderColor-Button-primary-outlined   → defaultThemeVars → $color-primary-600
borderColor-Button-secondary-outlined → (generated by generateButtonTones) → tone of $color-secondary
borderColor-Button-attention-outlined → (generated by generateButtonTones) → tone of $color-attention
```

These are set in [`xmlui/src/components/Button/Button.tsx:defaultThemeVars`](../../src/components/Button/Button.tsx) and derived dynamically in [`xmlui/src/components-core/theming/transformThemeVars.ts:generateButtonTones`](../../src/components-core/theming/transformThemeVars.ts).

**Input/form border:**

```
borderColor-Input-default   → root.ts → $colorSurface200
borderColor-Input           → TextBox.tsx defaultThemeVars → $borderColor-Input-default
borderColor-Select          → Select.module.scss → Input:borderColor-Select
```

The `Input:` prefix is a SCSS alias that makes `Select` inherit Input's token by default. All form components that use this system end up at `$colorSurface200`.

**The gap:**  
`$colorSurface200` and `$color-primary-600` are unrelated values. The only currently available workaround is hard-coding a raw color on `borderColor`, which bypasses the theme entirely.

### `borderColor` inline prop

`borderColor` is a layout prop recognised by `layout-resolver.ts` (`collectCss("borderColor", ...)`) and applied as an inline CSS style. It is purely a raw CSS override — it does not resolve against theme tokens, is not scoped to a component variant, and offers no semantic meaning.

## Implementation Steps

### Step 1: Add a semantic shared token `borderColor-outlined`

In [`xmlui/src/components-core/theming/themes/root.ts`](../../src/components-core/theming/themes/root.ts), add:

```ts
"borderColor-outlined": "$color-primary-500",
"borderColor-outlined--hover": "$color-primary-600",
"borderColor-outlined--active": "$color-primary-700",
"borderColor-outlined--focus": "$color-primary-600",
```

This gives the concept of "outlined border" a single named entry point in the theme that can be overridden globally. The primary color is the right default — it matches what `borderColor-Button-primary-outlined` already resolves to.

Add corresponding SCSS vars in [`xmlui/src/components-core/theming/_themes.scss`](../../src/components-core/theming/_themes.scss) following the existing pattern for `$borderColor`.

### Step 2: Wire Button's outlined token defaults to the new semantic token

In `Button.tsx` `defaultThemeVars`, change:

```ts
// Before:
[`borderColor-${COMP}-primary-outlined`]: "$color-primary-600",

// After:
[`borderColor-${COMP}-primary-outlined`]: "$borderColor-outlined",
[`borderColor-${COMP}-primary-outlined--hover`]: "$borderColor-outlined--hover",
[`borderColor-${COMP}-primary-outlined--active`]: "$borderColor-outlined--active",
```

This makes `Button`'s outlined primary border inherit from the semantic token. A theme that changes `borderColor-outlined` automatically updates `Button` too.

The `generateButtonTones` dynamic generator in `transformThemeVars.ts` runs after the explicit defaults and uses `??=` semantics, so the explicit defaults take precedence. No change needed there.

### Step 3: Add `variant` prop to `Select` (and other primary form inputs)

The `variant` prop for form controls means: "adjust border treatment to match the named Button variant". Initially support two values:

| Value | Behaviour |
|-------|-----------|
| `"default"` | Current: permanent border using `borderColor-Input` (grey, `$colorSurface200`) |
| `"outlined"` | New: uses the new semantic `$borderColor-outlined` token as the border color |

For `Select`, the SCSS change in `Select.module.scss` is:

```scss
// existing
$borderColor-Select: createThemeVar("Input:borderColor-#{$componentName}");

// add a variant-specific override that points at the new semantic token:
$borderColor-Select--outlined: createThemeVar("Input:borderColor-#{$componentName}--outlined");
```

In `Select.tsx`, add the `variant` prop and map `variant="outlined"` to inject the `--outlined` class (same pattern used for validation states like `--error`, `--warning`).

In `Select.tsx` `defaultThemeVars`, add:

```ts
[`borderColor-${COMP}--outlined`]: "$borderColor-outlined",
[`borderColor-${COMP}--outlined--hover`]: "$borderColor-outlined--hover",
[`borderColor-${COMP}--outlined--focus`]: "$borderColor-outlined--focus",
```

**Note:** The variant prop is additive, not a complete style overhaul. `outlined` only changes border color — the input keeps its standard background, padding, and typography. This is intentionally narrower than Button's variant (which also changes background).

Apply the same pattern to: `TextBox`, `AutoComplete`, `MultiSelect`, `NumberBox`, `DatePicker`, `TimePicker`, `Slider` (wherever a visible default border exists and derives from `borderColor-Input`). Each of these already uses the `Input:borderColor-*` inheritance chain, so the minimal change per component is: add `"outlined"` to the `variant` prop metadata, add `defaultThemeVars` entries pointing at `$borderColor-outlined`, and apply the class conditionally in the renderer.

### Step 4: Update `metadata` declarations

For each affected component, add `variant` to the component metadata if not already present, with available values including `"outlined"`. Use the existing `d()` helper pattern:

```ts
variant: d(
  "Controls the visual border treatment. 'outlined' matches the border color of an outlined Button.",
  [
    { label: "default", description: "Standard input border (surface color)" },
    { label: "outlined", description: "Accent border matching outlined Button" },
  ],
),
```

### Step 5: Surface the semantic token as a standard themeVar in the Select/Input components

In each affected component's `defaultThemeVars`, ensure the new outlined tokens appear in the `themeVars` schema so they are discoverable in the VS Code extension and docs tooling.

### Step 6: Documentation examples

Add an entry to the `Select` and `Button` component `.md` files showing the matched-border composition pattern:

```xml
<!-- Before: manual color match, brittle -->
<Select borderColor="blue" />
<Button variant="outlined" />

<!-- After: both inherit from borderColor-outlined -->
<Select variant="outlined" />
<Button variant="outlined" />
```

## Token Summary

| New token | Default value | Meaning |
|-----------|--------------|---------|
| `borderColor-outlined` | `$color-primary-500` | Default outlined border color |
| `borderColor-outlined--hover` | `$color-primary-600` | Hover state |
| `borderColor-outlined--active` | `$color-primary-700` | Active/pressed state |
| `borderColor-outlined--focus` | `$color-primary-600` | Keyboard focus |

These go in `root.ts` as part of the semantic token layer (same level as `borderColor`, `borderColor--disabled`, etc.).

## Backward Compatibility

- No existing `variant` value on form inputs is removed or changed. `variant` is a new prop on components that lacked it; existing markup without `variant` keeps the current grey Input border.
- The change to `borderColor-Button-primary-outlined` default (`$color-primary-600` → `$borderColor-outlined`) produces **no visual change** because `borderColor-outlined` is initialised to `$color-primary-500` — which is the same resolved color as `$color-primary-600` in the default theme. (Button's SCSS-generated tones would have yielded `$color-primary-600` before; after this change it yields `$color-primary-500`. There is a minor 1-step lightness difference. If this is undesirable, initialise `borderColor-outlined` to `$color-primary-600` instead.)
- Themes that already override `borderColor-Button-primary-outlined` explicitly still win (they are set after defaults, with no `??=`).

## Edge Cases & Open Questions

1. **`themeColor` prop on Button.** Button also accepts `themeColor` which modifies the palette for all variants. `secondary` and `attention` outlined buttons use their own color families. The new `borderColor-outlined` token represents the *primary* outlined color only. For `secondary` and `attention`, Button continues to use its own generated tones; those are not wired to `borderColor-outlined`. If a user wants to match a `secondary`-outlined Button with a form input, they can apply a local theme override — the plan does not attempt to solve the multi-color variant case initially.
2. **DropdownMenu component.** The request mentions DropdownMenu specifically. Verify whether it is a distinct component or an alias for Select. If distinct, apply the same `variant="outlined"` change.
3. **`borderColor` inline prop remains available.** It is still a valid escape hatch for one-off color values. The new `variant` system does not remove it. The two mechanisms stack: `variant` sets the CSS class; `borderColor` sets an inline style that takes precedence if provided.
4. **Dark theme.** `$color-primary-500` may need to be different in a dark theme context. The `borderColor-outlined` token is theme-overridable in dark themes just like all other root tokens — no special handling needed.
5. **Should `outlined` affect border-width and border-style too?** Button's `outlined` variant controls border-width, border-radius, and font as well. For form inputs these properties are typically set by the component's existing defaults and should not be overridden by `variant="outlined"`. Only `borderColor` (and related states) should change. Document this intentional asymmetry.
