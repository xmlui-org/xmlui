# `computedUses` — Огляд архітектури оптимізації

Документ описує механізми, які XMLUI використовує, щоб уникнути зайвих перемалювань
(re-renders) компонентів. Він написаний на рівні архітектури — без деталей реалізації.

---

## 1. Проблема

За замовчуванням React перемальовує весь піддерево щоразу, коли змінюється стан
батьківського компонента. В XMLUI-застосунку це означає, що навіть найдрібніша зміна
(наприклад, тік таймера) може перемалювати десятки компонентів, які насправді нічого
нового не відобразять.

```xml
<App var.tick="{0}" var.items="{loadItems()}">
  <Timer interval="{100}" onTick="tick++" />
  <Text value="Tick: {tick}" />

  <!-- Важка таблиця з тисячами рядків.
       Вона читає лише items — але без оптимізації
       перемалюється 10 разів на секунду разом із tick. -->
  <Table data="{items}" />
</App>
```

**Мета оптимізації:** автоматично визначити, від яких саме змінних залежить кожен
компонент, і перемальовувати його лише тоді, коли змінилась саме «його» змінна.

---

## 2. Два канали звуження

XMLUI розрізняє два незалежних типи змінних і звужує кожен окремо:

| Канал | Що звужується | Результат |
|---|---|---|
| `computedUses` | Локальний стан батьківського компонента | компонент не перемальовується при зміні «чужих» батьківських vars |
| `computedGlobalUses` | Змінні з `Globals.xs` | компонент не перемальовується при зміні «чужих» глобальних vars |

### 2.1 `computedUses` — звуження батьківського стану

```xml
<App var.tick="{0}" var.items="{loadItems()}">
  <Timer interval="{100}" onTick="tick++" />
  <Text value="Tick: {tick}" />

  <!-- computedUses = ["items"]
       Перемальовується лише коли змінюється items, не tick. -->
  <Table data="{items}" />
</App>
```

### 2.2 `computedGlobalUses` — звуження глобальних змінних

```xml
<!-- Globals.xs -->
var.sortBy = "name"
var.view   = "table"
var.events = {}
```

```xml
<!-- FilesPage.xmlui -->
<Fragment>
  <!-- computedGlobalUses = ["sortBy"]
       Не перемальовується при зміні view або events. -->
  <Table sortBy="{sortBy}" data="{fileEntries}" />

  <!-- computedGlobalUses = ["view"]
       Не перемальовується при зміні sortBy або events. -->
  <TileGrid when="{view === 'icons'}" data="{fileEntries}" />
</Fragment>
```

## 3. Як визначаються залежності — статичний аналіз

Фреймворк аналізує розмітку **один раз під час завантаження** (до першого рендеру),
обходячи дерево компонентів знизу вгору. Для кожного компонента-контейнера він збирає
мінімальний список змінних, від яких той справді залежить.

### 3.1 Що аналізується

- значення props: `data="{items}"`, `when="{view === 'table'}"`
- обробники подій: `onClick="doSort()"`, `onLoaded="(d) => { files = d; }"`
- власні vars компонента та їхні вирази-ініціалізатори
- функції в `.xs`-файлах та `<script>`-блоках — **рекурсивно** (якщо функція викликає
  іншу локальну функцію, аналізатор заходить і в неї)

### 3.2 Що фільтрується і не потрапляє в залежності

| Що | Приклад | Чому |
|---|---|---|
| JS-стандарт | `Math.round()`, `JSON.stringify()`, `Array.from()` | Ніколи не є станом XMLUI |
| Браузерні глобали | `window.location`, `document.title` | Не є частиною стану компонента |
| Фреймворкові утиліти | `Actions`, `toast`, `navigate`, `Log` | Вбудовані в scope усіх виразів |
| Лексичні змінні | `$item`, `$index`, `$context` | Інжектуються фреймворком в ітераторах |
| Власні vars компонента | `var.x` оголошений у самому компоненті | Локальний, не приходить згори |

```xml
<List data="{items}">
  <!-- $item і $index — лексичні, не залежності від батька -->
  <Text value="{$index}: {$item.name}" />
  <!-- JSON — стандартна JS-функція, не стан -->
  <Text value="{JSON.stringify($item)}" />
</List>
```

## 4. Важкі компоненти — автоматичне загортання

Компоненти типу `Table`, `Select`, `List`, `TileGrid`, `Tree`, `DataGrid` позначені як
«важкі». Якщо такий компонент читає змінні батька, фреймворк **автоматично** загортає
його в ізолюючий контейнер із `computedUses`. Якщо не читає нічого — ізоляція не
потрібна і зайва обгортка не створюється.

```xml
<App var.filter="{''}" var.theme="{'light'}">

  <!-- Читає filter → автоматично отримує computedUses=["filter"].
       Не перемальовується при зміні theme. -->
  <Select data="{filteredItems(filter)}" />

  <!-- Нічого не читає з батька → обгортка не створюється. -->
  <Select>
    <Option value="a" />
    <Option value="b" />
  </Select>

</App>
```

## 5. Аналіз функцій у `.xs`-файлах

Коли логіка виражена не в розмітці, а у функціях `.xs`/`<script>`, аналізатор
**заходить у тіла цих функцій** і витягує звідти реальні залежності.

```js
// FilesView.xmlui.xs
function rowClass(item) {
  // Читає глобал catalogSelection — аналізатор це бачить
  return item.id === catalogSelection.activeId ? 'selected' : '';
}
```

```xml
<!-- FilesView.xmlui -->
<Fragment>            <!-- ← власник функції rowClass -->
  <Table>
    <Column cellRenderer="{(item) => rowClass(item)}" />
  </Table>
</Fragment>
```

`Fragment` як власник `rowClass` отримає `computedGlobalUses = ["catalogSelection"]` —
правильно, бо аналізатор рекурсивно пройшов тіло функції.

Рекурсія безпечна: якщо функція `A` викликає `B`, а `B` — знову `A` (взаємна рекурсія),
механізм «відвіданих вузлів» запобігає нескінченному циклу.

## 6. Механізм UID-ескейпінгу

Коли компонент оголошує `id`, він реєструє своє API у **найближчому** контейнері-власнику.
При звуженні фреймворк стежить, щоб цей UID завжди залишався доступним для тих, хто на
нього посилається.

```xml
<App>
  <Fragment var.x="{0}">   <!-- контейнер, але не явний власник -->
    <Select id="mySelect" />

    <!-- mySelect.value — посилання на API сусіда.
         Фреймворк автоматично включить "mySelect" у computedUses Fragment-а,
         щоб Select.value не зник зі звуженого стану. -->
    <Button
      label="{mySelect.value}"
      onClick="submit(mySelect.value)"
    />
  </Fragment>
</App>
```

## 7. Як звуження захищає обробники подій

Звуження стосується лише **рендерингу**: що компонент відображає. Але обробник події
може писати в будь-яку змінну, навіть ту, що не відображається.

```xml
<Fragment var.counter="{0}">
  <!-- counter — у computedUses, компонент його відображає -->
  <Text value="{counter}" />

  <!-- hidden — НЕ у computedUses (компонент його не читає),
       але запис все одно спрацює коректно -->
  <Button onClick="hidden = Date.now()" />
</Fragment>
```

Фреймворк передає повний стан обробникам через стабільний ref — тому запис у «незвужену»
змінну завжди доходить до мети, але не провокує зайвих рендерів.

## 8. Поточні обмеження

Звуження вимикається консервативно в кількох ситуаціях, де аналізатор не може
**достовірно** встановити залежності:

### 8.1 Важкий компонент викликає функцію батьківського компонента

Аналізатор знає, які глобали читає функція `rowClass` у свого власника (`Fragment`).
Але `Table` — важкий, формує окремий контейнер — отримує лише **ім'я** функції, без
списку її залежностей. Тому `Table` залишається консервативним (без `computedGlobalUses`).

```xml
<Fragment>             <!-- власник rowClass — знає що читає catalogSelection ✅ -->
  <script>
    function rowClass(item) { return item.id === catalogSelection.activeId ? 'sel' : ''; }
  </script>

  <!-- Table — важкий, бачить лише ім'я "rowClass", не знає про catalogSelection.
       Звуження глобалів для Table вимкнено. ❌ -->
  <Table>
    <Column cellRenderer="{(item) => rowClass(item)}" />
  </Table>
</Fragment>
```

**Планується:** передавати список глобальних залежностей функції разом із її ім'ям, щоб
`Table` отримував `computedGlobalUses = ["catalogSelection"]` автоматично.

### 8.2 Функція імпортована з іншого `.xs`-файлу

Якщо компонент викликає функцію, імпортовану з сусіднього `.xs`, аналізатор не заходить
у той файл і не бачить, які глобали вона читає всередині. Звуження вимикається.

```xml
<Fragment>
  <script>
    import { publishEvent } from "../shared.xs";
  </script>

  <!-- publishEvent читає events — але аналізатор не знає про це.
       Звуження глобалів для Fragment вимкнено. ❌ -->
  <Button onClick="publishEvent('file:deleted', item)" />
</Fragment>
```

**Планується:** резолвити `.xs`-імпорти під час аналізу і діставати залежності з тіл
імпортованих функцій.

### 8.3 Глобальні функції `Globals.xs` завжди передаються повністю

Функції, оголошені в `Globals.xs`, завжди включаються у звужений об'єкт глобалів —
навіть якщо конкретний компонент їх не викликає. Змінні при цьому звужуються коректно.

**Планується:** відстежувати, які саме глобальні функції викликає компонент, і передавати
лише їх.

## 9. Порядок виконання аналізу

```
1. Vite-плагін (збірка / dev-server) — перший прохід
   Аналізує кожен .xmlui-файл одразу після парсингу.
   Globals.xs ще невідомий → computedGlobalUses ще не встановлюється.

2. StandaloneApp (завантаження в браузері) — авторитетний прохід
   Доєднуються .xs code-behind функції та відомий Globals.xs.
   computedUses і computedGlobalUses обчислюються остаточно — ДО першого рендеру.

3. (За потреби) Після резолву імпортів — якщо .xs-файли завантажувались асинхронно,
   залежності перераховуються ще раз перед повноцінним рендером.
```

Авторитетний прохід перезаписує результат першого, тому кінцевий результат завжди
коректний.

---

## 10. Коротке резюме

| Механізм | Що робить |
|---|---|
| `computedUses` | Звужує локальний стан батька — компонент не бачить «чужі» vars |
| `computedGlobalUses` | Звужує глобальні змінні — компонент не реагує на «чужі» глобали |
| Автоматичне загортання важких компонентів | `Table`/`Select`/`List` тощо отримують ізоляцію без явного `uses` |
| Аналіз функцій `.xs` (рекурсивний) | Залежності витягуються навіть із глибоко вкладених викликів |
| UID-ескейпінг | Звуження не «відрізає» компонент від API його сусідів |
| Повний стан у обробниках | Запис у будь-яку змінну завжди спрацьовує, навіть якщо вона не у `computedUses` |
