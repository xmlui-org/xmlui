# Website Style Fix Learnings - 2026-07-14

- Old/new website comparison should inspect computed theme variables first. The reported larger logo/search, constrained header divider, and search input styling deviation all correlated with the new page falling back to default theme vars instead of `xmlui-website-theme`.
- The website config is exported from `/src/config.ts` and declares `defaultTheme: "xmlui-website-theme"` plus the custom theme list. Theme modules also export `resources`, so broad config detection can mistake a theme definition for app config unless theme-shaped objects are excluded.
- In dev, Vite may keep serving a stale optimized `xmlui` dependency after framework source edits. If computed vars still show defaults (`--xmlui-height-AppHeader: 3.5em`, `--xmlui-maxWidth-content-AppHeader: 1280px`) after a core theme fix, restart the new website dev server or force dependency re-optimization before rechecking DOM.
- Runtime startup must carry standalone config (`name`, `appGlobals`, `themes`, `defaultTheme`, resources) into `XmluiRoot`; old XMLUI treated these as part of the app contract. Website startup can pass `/src/config.ts` explicitly to avoid ambiguous runtime scanning.
- Extension components imported from public `xmlui` expect themed authoring exports (`Button`, `TextBox`, etc.). Bare React exports lose the generated theme class and produce size/spacing drift in packages such as `xmlui-search`.
- `XmluiRoot` must include extension theme metadata and extension theme definitions. The website uses docs/search extension components, so core-only theme metadata misses extension defaults.
- Nested XMLUI apps on the website can mount their own providers; they must not emit document-level CSS variables or they overwrite the host website theme on `<html>`.
- The root light theme aliases for `color-surface-100` and `color-surface-600` must point at `const-*` variables, matching `/Users/dotneteer/source/xmlui/xmlui/src/components-core/theming/themes/root.ts`. Website themes that set `color-surface` rely on those aliases to pick up regenerated surface tones; otherwise copy in `HeadLines`/`Benefits` falls back to the default XMLUI blue-gray `rgb(71, 108, 133)` instead of the old website gray `rgb(99, 98, 106)`.
- In runtime `AppHeader`, avoid passing a default empty `ProfileMenu` element when there is no logged-in user; otherwise an empty right-items slot reserves header flex gap and shifts search left.
- Legacy `ComponentDef.props` is optional, while compiled runtime `XmluiElement.props` is a string map. Core fixes that bridge old and new renderers should use a local `props ?? {}` fallback and stringify runtime layout-context values before merging them into compiled props.
- `AppReact` bridges legacy and runtime app context APIs. Keep navigation handlers in the runtime object shape, type nullable DOM slots explicitly, and only render context-aware header helpers when a `renderChild` callback is available.
- App layout compatibility rules belong in `App.module.scss`, not injected `<style>` elements. The horizontal band and Pages width rules can be expressed through the App slot classes (`headerWrapper`, `footerWrapper`, `navPanelWrapper`) plus `:global(.xmlui-Pages)`.
- The active runtime provider/compiler are now named `XmluiThemeProvider` and `themeCompiler`/`compileThemeModel`. The shadow diagnostic and canary test scaffolding were migration-only; the retained tests now cover provider/compiler behavior directly.
- Formatted inline text markup must restore collapsed source whitespace around nested inline elements. Old XMLUI rendered `TextNode` values through `/Users/dotneteer/source/xmlui/xmlui/src/components-core/rendering/renderChild.tsx`; rewrite runtime rendering now uses text segment ranges for `text -> element` boundaries, and the `Text` renderer inserts scoped synthetic spaces for formatted `element -> element` children such as nested `<Text>` followed by `<Slot>`. Verified with `npm --workspace xmlui run test -- Text.spec.tsx renderer.spec.tsx`.
- Runtime `App` must inspect the actual `Footer` child, including when the shell child is wrapped by a transparent `Theme`, and pass `footerSticky` into `AppReact`; otherwise `sticky="{false}"` is visible on the footer node but the app footer slot remains docked. Runtime `Footer` padding props belong on the content wrapper, matching old `/Users/dotneteer/source/xmlui/xmlui/src/components/Footer/Footer.tsx` behavior, while the outer shell carries border/background/role. Runtime `ContentSeparator` size props must resolve theme tokens such as `$space-3_5`; otherwise website footer separators render with `height: 0`.
- Runtime `Markdown` must forward `codeHighlighter` into `MarkdownReact`, matching old `/Users/dotneteer/source/xmlui/xmlui/src/components/Markdown/Markdown.tsx`; otherwise explicit XMLUI code fence highlighting props are ignored. Both runtime and React Markdown indent removal must use the old minimum-indent algorithm from `/Users/dotneteer/source/xmlui/xmlui/src/components/Markdown/MarkdownReact.tsx`; the migrated "smallest positive indent" fallback strips indentation from fenced code lines when the fence itself starts at column zero. Markdown's custom `code` renderer must preserve `data-language`/meta props so `parseMetaAndHighlightCode` can detect `xmlui` fences.
- Documentation syntax highlighting depends on the public `xmlui/syntax/textmate` export. The rewrite must export the real old TextMate grammar/theme JSON from `/Users/dotneteer/source/xmlui/xmlui/src/syntax/textMate`, not a placeholder grammar, and the grammar must include the `xmlui-pg` alias so docs playground fences can be highlighted through the same XMLUI grammar.
- Runtime `Stack` must only pass click/context-menu/mounted event props when the XMLUI node declares those events. Passing an unconditional wrapper around `adapter.event("click")` creates a no-op React `onClick`, which turns on Stack's `_handlesClick` cursor class and makes nested Markdown text inherit `cursor: pointer` across the docs layout. Old XMLUI forwarded `props.onClick`, so absent markup stayed absent.
- Markdown indent removal needs to distinguish XMLUI authoring indentation from fenced-code body indentation. When the first rendered markdown line is unindented but the rest of the XMLUI child text carries a common source indent, use the positive authoring indent from non-fence lines and fence delimiter lines, then strip that indent from all lines. Do not let indented fenced-code body lines alone choose the indent, or column-zero fences lose their intentional leading spaces.
- Nested app frames should inherit shell text color from the host website unless `textColor-NestedApp` is explicitly themed. A hard `#000` fallback makes website code-fence headers and frame controls drift. The rendered preview uses normal `App` layout rules, so horizontal layouts must clip horizontal overflow; otherwise wide sample rows in fixed-height `xmlui-pg` frames produce visible nested app scrollbars.
- The `Pick a theme` playground path uses Markdown `xmlui-pg` -> `AppWithCodeView` -> nested app. Measuring old (`localhost:5176`) and new (`localhost:5174`) local dev servers after scrolling the sample into view showed the old App reserves `scrollbar-gutter: stable both-edges`: page content starts 15px inside the nested App before its own 15px App padding and the sample's 15px `VStack` padding, so the "Theme" label is 46px from the frame edge. The rewrite must preserve that reserved gutter while using `overflow-x: hidden` to avoid the visible horizontal scrollbar and `min-height: 0` to avoid the vertical overflow. Nested root themes must also scope compiled CSS variables to the nested subtree when document-level theme mutation is disabled; otherwise custom nested theme vars such as `paddingHorizontal-content-App` do not affect the preview.
