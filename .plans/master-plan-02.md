# XMLUI Rewrite Master Plan

Status: first draft  
Source baseline: `/Users/dotneteer/source/xmlui`

## 1. Purpose

Re-implement XMLUI while preserving the current user and developer experience:
markup, scripting semantics, components, themes, routing, data operations,
forms, extensions, docs generation, CLI tools, VS Code support, playground,
build outputs, integration tests, and deployment behavior.

The rewrite is motivated by architecture simplification and performance. The
current codebase is maintainable, but it evolved through many feature waves and
now contains concepts, layers, and runtime mechanisms that can be replaced by a
compiler-centered design.

Primary performance target: make expression evaluation and event handler
execution at least 50 times faster than the current interpreted model, while
remaining compatible with XMLUI's existing scripting semantics.

## 2. Compatibility Contract

The old project remains the source of truth. Compatibility must cover:

- `.xmlui`, `.xmlui.xs`, `.xmlui.xm`, `Globals.xs`, `config.json`, and
  `config.ts` behavior;
- standalone/buildless apps, Vite dev apps, production builds, extension
  packages, and SSG output;
- built-in components, user-defined components, slots, templates, metadata,
  theme variables, behaviors, component APIs, and context variables;
- expression and event semantics, including XMLUI's JavaScript-like sandbox,
  optional member access defaults, reactive `var`, async handler behavior,
  action APIs, and diagnostics;
- CLI commands, package exports, generated metadata, docs generation, VS Code
  language services, and integration-test app behavior.

Any intentional incompatibility must be explicit, planned, and justified.

## 3. Old Architecture Findings

The current XMLUI runtime parses markup into `ComponentDef` trees, renders them
through React, and evaluates expressions and handlers through custom AST
interpreters. Both standalone and Vite modes converge on the same runtime
rendering pipeline; Vite pre-bundles `.xmlui` files but still leaves expression
and event execution interpreted at runtime.

State is organized around implicit and explicit containers. Each stateful node
uses a `StateContainer` plus `Container` pair, merging parent state, reducer
state, component APIs, context variables, local variables, and globals into one
flat state object. Mutations are detected through proxies and routed to the
owning container reducer.

The main performance costs and simplification opportunities are:

- runtime interpretation of binding expressions and event handlers;
- proxy-based mutation tracking;
- broad container/state merging and repeated subtree rendering;
- runtime node transformations that could be compiled earlier;
- generated API-bound handler strings that are interpreted again;
- duplicated concerns across runtime, Vite transform, metadata generation,
  language server, docs generation, and SSG.

The old implementation already contains useful stepping stones: dependency
analysis, `computedUses`, reactive cycle detection, metadata-driven contracts,
render-count profiling, SSG prerendering, and detailed language-server metadata.

## 4. Target Architecture

The new architecture should be compiler-first:

- parse XMLUI markup, code-behind files, and config into a typed intermediate
  representation;
- compile binding expressions into JavaScript functions;
- compile event handlers and action adapters into JavaScript functions;
- derive a static reactive dependency graph for variables, props, loaders,
  handlers, context variables, and component APIs;
- replace proxy mutation tracking with explicit invalidation and update
  scheduling generated by the compiler, following the Svelte-style model where
  assignments trigger known reactive work;
- keep a small runtime responsible for component mounting, scheduling, data
  fetching, routing, theming, extension registration, diagnostics, and hydration.

The compiler should be shared by standalone runtime compilation, Vite dev
builds, production builds, metadata generation, language services, docs
generation, SSG, and compatibility testing.

## 5. Execution Models

### Standalone App

An `index.html` loads a single XMLUI framework bundle. The framework fetches
`Main.xmlui`, discovers referenced components, loads all required `.xmlui`,
`.xs`, config, theme, and extension resources, compiles them in the browser,
and starts the app.

Compatibility priority: preserve buildless static hosting and the existing app
folder model.

### Vite Dev Build

The Vite dev server loads XMLUI source files, compiles them to debuggable
JavaScript modules, emits source maps back to XMLUI source locations, supports
HMR, and starts the app.

Compatibility priority: preserve current Vite app ergonomics while making
browser debugging of XMLUI expressions and handlers practical.

### Production Build

The build performs all possible parsing, compilation, dependency analysis,
metadata extraction, route discovery, asset preparation, and optimization in
advance. Startup should mostly instantiate precompiled modules and hydrate
data/routing/theme state.

Compatibility priority: preserve package exports, CLI behavior, generated
artifact shape, and extension package contracts.

### Static Site Generation

SSG should compile and prerender XMLUI apps into static HTML pages that display
correct content and styling before JavaScript loads. Hydration should attach the
compiled runtime with minimal visible work.

Compatibility priority: preserve current SSG route discovery, fallback behavior,
search-index generation, CSS injection guarantees, and `preview-ssg` behavior.

## 6. First Migration Experiment

The first executable target is a deliberately small Vite dev-server slice. It
must run three tiny XMLUI apps correctly before any broader framework work.

### Scope

Build only enough infrastructure for Vite dev mode to support:

- `Main.xmlui` as the app entry point;
- user-defined components in sibling `.xmlui` files;
- `<App>`, `<H1>`, and `<Button>`;
- text nodes with embedded expressions;
- literal props and expression props;
- `$props` for user-defined component inputs;
- local variables declared with `var.name="{expr}"`;
- global variables declared with `global.name="{expr}"`;
- lexical shadowing where a local variable hides a global variable;
- event handlers compiled from simple XMLUI script, starting with `count++`;
- incremental UI updates after event-driven state changes.

This slice intentionally excludes routing, data fetching, forms, theming,
extension packages, code-behind files, production builds, standalone mode, SSG,
language-server behavior, and full XMLUI scripting coverage.

The slice is not valid without tests. It must be supported by both unit and
E2E tests before it can be considered complete.

### Target Apps

Example 1 validates local component state:

```xml
<App var.count="{0}">
  <H1>Counter example</H1>
  <Button onClick="count++">
    Click to increment: {count}
  </Button>
</App>
```

Example 2 validates user-defined components, isolated local state, default
props, and repeated component instances:

```xml
<!-- Main.xmlui -->
<App>
  <H1>Counter with components</H1>
  <IncrementButton />
  <IncrementButton label="Counter #2" />
  <IncrementButton label="Counter #3" />
</App>
```

```xml
<!-- IncrementButton.xmlui -->
<Component name="IncrementButton" var.count="{0}">
  <Button onClick="count++">
    {$props.label || 'Click to increment'}: {count}
  </Button>
</Component>
```

Example 3 validates global state, shared updates across component instances,
and local shadowing:

```xml
<!-- Main.xmlui -->
<App global.count="{0}">
  <H1>Counter with components</H1>
  <IncrementButton />
  <IncrementButton label="Global Counter #2" />
  <IncrementButton label="Global Counter #3" />
  <Button var.count="{0}" onClick="count++">
    Local count: {count}
  </Button>
</App>
```

```xml
<!-- IncrementButton.xmlui -->
<Component name="IncrementButton">
  <Button onClick="count++">
    {$props.label || 'Click to increment (Global)'}: {count}
  </Button>
</Component>
```

### Implementation Shape

The first implementation should create the smallest vertical path:

1. a minimal Vite project and XMLUI plugin that transforms `.xmlui` files;
2. a parser path that can read the target markup and preserve source locations;
3. a compiler IR for elements, text, props, variables, globals, events, and
   user-defined components;
4. generated JavaScript modules for `Main.xmlui` and component files;
5. compiled expression functions for `{0}`, `{count}`, `$props.label || ...`,
   and mixed text/expression content;
6. compiled event functions for `count++`;
7. a tiny runtime with component registration, instance-local state,
   app-global state, lexical name resolution, shadowing, and invalidation;
8. React renderers for `App`, `H1`, and `Button`;
9. a browser verification path that clicks each button and checks visible
   counter updates.

The generated code should favor explicit state slots and explicit invalidation
over proxy mutation tracking, even in this tiny prototype.

### Required Architecture Pieces

This experiment should borrow ideas from the old architecture, not copy its
layers wholesale. The goal is to create a small but real vertical slice that can
grow into the full compiler/runtime later.

#### 1. Project Skeleton

Required now:

- root `package.json` with scripts for `dev`, `build` or `check`, and test
  verification when available;
- TypeScript configuration;
- Vite configuration that installs the new XMLUI plugin;
- a minimal example app folder containing `index.html`, `src/main.tsx`,
  `src/Main.xmlui`, and optional sibling component files such as
  `src/IncrementButton.xmlui`;
- source folders for `compiler`, `runtime`, `vite-plugin`, and `examples` or
  test fixtures.

Old XMLUI source to learn from:

- root workspace scripts;
- `xmlui/src/nodejs/vite-xmlui-plugin.ts`;
- Vite app entry patterns in website/playground/integration-test apps.

Keep out for now:

- npm workspaces;
- Turborepo;
- CLI packaging;
- create-app templates;
- standalone bundle packaging;
- extension package builds.

#### 2. Vite Dev-Server Integration

Required now:

- transform `.xmlui` files into JavaScript/TypeScript modules;
- distinguish `Main.xmlui` from `<Component name="...">` files;
- let `src/main.tsx` import `Main.xmlui` and start the runtime;
- make sibling user-defined components available to `Main.xmlui`;
- return usable source maps or at least preserve source ranges in compiled
  metadata for later source-map work.

Old XMLUI source to learn from:

- the old Vite plugin's basic file-extension dispatch and module transform;
- its use of parsed source as the shared artifact between build, diagnostics,
  and runtime.

Simplify:

- do not emit old `ComponentDef` data modules;
- do not run analyzer, accessibility, type-contract, reactive-cycle,
  code-behind, module import, DataSource, or metadata passes yet.

#### 3. Markup Parser

Required now:

- parse enough XML to handle elements, attributes, comments, self-closing tags,
  text nodes, and nested children;
- preserve source locations for elements, attributes, text, expressions, and
  event handlers;
- understand the special root tags `<App>` and `<Component>`;
- split attributes by prefix: normal props, `var.*`, `global.*`, and `on*`
  event handlers;
- parse text nodes with mixed literal text and `{expression}` segments.

Old XMLUI source to learn from:

- `createXmlUiParser`;
- `nodeToComponentDef`;
- compound component collection from `<Component>`.

Simplify:

- no `<property>`, `<event>`, `<variable>`, `<global>`, `<Slot>`, CDATA,
  namespaces, loaders, `when`, responsive props, layout props, or parser-level
  diagnostics beyond clear syntax failures needed for the examples.

#### 4. Expression and Event Parsing

Required now:

- parse the expression subset used by the examples:
  - numeric literals;
  - identifier reads;
  - `$props.member` reads;
  - string literals;
  - logical OR (`||`);
  - postfix increment (`count++`);
- identify read dependencies and write targets;
- reject unresolved assignment targets early when practical.

Old XMLUI source to learn from:

- scripting parser;
- dependency collection;
- XMLUI's default optional member access semantics.

Simplify:

- the first compiler may use a small dedicated parser or a JavaScript parser
  library, but it should expose XMLUI-specific semantics through our own IR;
- no async handlers, statement queues, imports, functions, arrow callbacks,
  member assignment, array/object literals, banned globals, timeouts, or full
  sandbox enforcement yet.

Managed React feature kept now:

- expressions and handlers are not raw application JavaScript strings executed
  ad hoc; they are parsed, checked against XMLUI scope, and compiled through the
  framework.

#### 5. Compiler IR

Required now:

- represent `App`, built-in elements, text parts, user-defined component
  definitions, props, local variables, globals, and events;
- represent lexical scopes explicitly;
- represent expression reads and event writes as dependency metadata;
- mark whether a binding depends on local state, global state, or `$props`.

Old XMLUI source to learn from:

- `ComponentDef`;
- `CompoundComponentDef`;
- `computedUses`;
- reactive graph plans.

Simplify:

- do not carry old renderer descriptors, behavior metadata, loader arrays,
  component APIs, action registry, theme parts, or layout context.

Managed React feature kept now:

- the IR is the contract boundary. Later editor diagnostics, runtime checks,
  docs, and builds should reuse this artifact instead of inventing parallel
  models.

#### 6. Code Generation

Required now:

- generate modules that export compiled component definitions or component
  factories;
- compile expressions into JavaScript functions with explicit parameters, such
  as local state, global state, and props;
- compile `count++` into a write to the resolved state slot plus invalidation;
- generate render functions that call the tiny runtime instead of interpreting
  expression strings at render time.

Old XMLUI source to learn from:

- Vite transform output shape;
- `extractParam` only as a compatibility reference for expression results.

Simplify:

- no old AST interpreter in the hot path;
- no proxy wrapping;
- no generated action strings that are fed back into a scripting interpreter.

#### 7. Runtime State Model

Required now:

- app-global store owned by the root app instance;
- component-instance store for each `App` or user-defined component instance
  that declares local `var.*`;
- lexical lookup rule: local variable first, then global variable, then `$props`
  where applicable;
- local shadowing over globals;
- isolated local state for repeated user-defined component instances;
- shared global state across component instances;
- explicit invalidation subscriptions for compiled bindings.

Old XMLUI source to learn from:

- state layer ordering;
- mutation routing;
- user-defined component scope isolation;
- global variable behavior.

Simplify:

- no `StateContainer`/`Container` pair;
- no reducer action enum;
- no immer;
- no component APIs;
- no context variables other than `$props`;
- no parent `uses` boundaries yet.

Managed React feature kept now:

- authors never touch React state directly. XMLUI owns state slots, mutations,
  invalidation, and re-rendering.

#### 8. Rendering Pipeline

Required now:

- a tiny recursive renderer for compiled nodes;
- built-in renderers for `App`, `H1`, and `Button`;
- text rendering that recomputes only the bindings affected by an invalidated
  state slot, or at least records the dependency granularity needed to get
  there;
- user-defined component instantiation with resolved props and isolated local
  state;
- event handler attachment for `Button` `onClick`.

Old XMLUI source to learn from:

- `renderChild`;
- `ComponentWrapper`;
- `CompoundComponent`.

Simplify:

- no provider stack beyond React root and the tiny XMLUI runtime context;
- no behavior chain;
- no theming;
- no slots;
- no `when`;
- no error boundaries unless a minimal developer-facing parse/render error is
  cheap to add.

#### 9. Minimal Managed React Contracts

Required now:

- component names and allowed props/events for `App`, `H1`, `Button`, and
  user-defined components should be known to the compiler, even if represented
  as a tiny inline registry;
- unknown component or unknown event errors should be reported clearly;
- local/global variable declarations should be validated enough to prevent
  silent unresolved writes;
- source locations should be carried with diagnostics.

Old XMLUI source to learn from:

- component metadata;
- type-contract diagnostics;
- language-server metadata shape.

Simplify:

- no generated docs metadata;
- no full prop type coercion;
- no strict modes;
- no accessibility, lifecycle, concurrency, i18n, DOM sandbox, audit logging,
  or versioning diagnostics in this experiment.

Managed React features deliberately deferred:

- lifecycle hooks and automatic cleanup;
- declarative data fetching;
- resource cancellation;
- structured errors;
- deterministic handler scheduling;
- DOM API sandboxing;
- accessibility enforcement;
- theme sandboxing;
- localization;
- release/versioning contracts.

#### 10. Verification Harness

Required now:

- a dev-server example for each target app;
- browser automation that clicks buttons and checks visible text;
- a compatibility note that maps each implemented behavior to the old XMLUI
  semantics and lists deferred behavior;
- a unit-test harness for parser, compiler IR, dependency extraction, code
  generation, and runtime state rules;
- an E2E test harness that can run XMLUI snippets and sibling component
  definitions with nearly the same test bodies as the old suite.

Old XMLUI source to learn from:

- integration-test app structure;
- Playwright-style user-visible assertions;
- render-count profiling as a future benchmark idea;
- `xmlui/src/testing/fixtures.ts`, especially the `initTestBed` testing style;
- old E2E tests for globals, shadowing, compound components, and binding
  updates;
- old parser/transform unit tests for variable and global declarations.

Simplify:

- no full old test-bed infrastructure;
- no cross-browser matrix;
- no production build verification until the Vite dev slice is stable.

### Test Reuse Contract

Existing XMLUI unit and E2E test cases are compatibility assets. The rewrite may
replace the test infrastructure, but should keep the test cases themselves
literally the same whenever practical.

Required approach:

- preserve old XMLUI snippets, expected visible text, and interaction sequences
  as much as possible;
- allow edits to imports, fixtures, helper setup, and app mounting only where
  the new infrastructure requires it;
- prefer adapting old tests to the new harness over rewriting their intent in a
  new style;
- when a test cannot be used yet because the feature is out of scope, mark it
  as deferred and record the missing feature rather than deleting the case;
- build the first experiment's tests from the old cases covering:
  - local `var` state and `count++`;
  - global variables;
  - local shadowing over globals;
  - user-defined components;
  - repeated component instances with isolated local state;
  - repeated component instances sharing global state;
  - parser handling of `var.*`, `global.*`, text expressions, props, and
    `<Component name="...">`.

The initial harness should provide equivalents for:

- `initTestBed(markup, { components })` for E2E tests;
- Playwright `page` and `expect` assertions;
- unit helpers that parse XMLUI source and expose the new compiler IR instead
  of the old `ComponentDef`.

The first experiment succeeds only when the three target apps pass both:

- focused unit tests proving parse, scope, dependency, and code-generation
  behavior; and
- focused E2E tests proving the actual Vite dev-server runtime behavior in the
  browser.

### Compatibility Checks

The experiment succeeds when:

- Example 1 increments one local counter.
- Example 2 renders three independent counters; clicking one does not update
  the others.
- Example 3 renders three buttons sharing one global counter; clicking any of
  those updates all global-counter displays, while the local button keeps its
  own independent count.
- The same XMLUI source shape can be served through the Vite dev server without
  a manual build step.
- Unit tests and E2E tests, adapted from the old XMLUI suite with minimal case
  changes, pass for the implemented slice.
- The implementation records which old XMLUI semantics were intentionally
  implemented and which were deferred.

## 7. Incremental Experiment Roadmap

Assuming the first Vite counter experiment succeeds, continue with small
experiments that each teach one important architectural lesson. Every experiment
must add compatible behavior, preserve or port old tests with minimal changes,
and avoid pulling in broad framework machinery before it is needed.

### Experiment 2: Broader Expression Compilation

Scope:

- expand expression support from counters to object/array literals, arithmetic,
  comparisons, ternaries, nullish coalescing, optional member access, function
  calls, and basic arrow callbacks;
- preserve XMLUI's default optional member access behavior;
- compile expressions to JavaScript functions while keeping dependency metadata.

What we learn:

- whether the expression compiler can match old XMLUI semantics without falling
  back to the interpreter;
- how source maps and diagnostics should point from generated JavaScript back to
  XMLUI expressions;
- where native JavaScript behavior must be wrapped to preserve XMLUI-specific
  semantics.

### Experiment 3: Handler Compilation and Mutation Semantics

Scope:

- expand event handlers from `count++` to assignments, compound assignments,
  blocks, conditionals, loops, local `let`/`const`, function calls, and simple
  multi-statement handlers;
- keep state writes explicit and compiler-routed;
- preserve statement-to-statement visibility of state updates where old XMLUI
  promises it.

What we learn:

- whether compiled handlers can replace the statement queue without losing
  compatibility;
- how to model handler scheduling, errors, and invalidation without the old
  container/reducer machinery;
- which scripting features need XMLUI-specific runtime helpers.

### Experiment 4: Reactive Graph and Derived Variables

Scope:

- support reactive `var` expressions depending on other vars, globals, props,
  and component APIs when available;
- implement dependency graph invalidation for derived values;
- add cycle detection and diagnostics for the implemented subset.

What we learn:

- whether Svelte-style compile-time dependency analysis can replace proxy-based
  change detection;
- how to handle the old "reactive until assigned" variable rule;
- how much runtime graph bookkeeping remains necessary after compilation.

### Experiment 5: User-Defined Component Composition

Scope:

- grow UDC support beyond props and local vars to default children, slots,
  template properties, event emission, and component methods;
- preserve UDC scope isolation: parent variables are not visible unless passed
  through props or slots.

What we learn:

- how to compile component boundaries without recreating the old compound
  component/container stack;
- how slots should carry parent scope and component-provided context variables;
- whether UDC metadata can be generated directly from the compiler IR.

### Experiment 6: Built-In Component Expansion

Scope:

- add enough built-in components to run a small realistic app: `Text`, `Stack`,
  `HStack`, `VStack`, `Items`, `TextBox`, `Checkbox`, `Select`, and basic
  form-free input state;
- introduce minimal metadata for props, events, and exposed component APIs.

What we learn:

- how generic the renderer/runtime boundary is across visual, layout, list, and
  input components;
- where component APIs fit into compiled scope and invalidation;
- how much of the old behavior chain can be replaced by compile-time metadata
  and simple runtime wrappers.

### Experiment 7: Data Operations and Managed Fetching

Scope:

- implement a minimal `DataSource`, `APICall`, `Actions.callApi`, loader state,
  cache invalidation, request cancellation, and error surface;
- run old tests that cover basic data binding and imperative API calls.

What we learn:

- how declarative data flow should connect to the compiler's reactive graph;
- whether TanStack Query remains the right runtime dependency or should be
  wrapped behind a smaller contract;
- how Managed React resource lifecycle guarantees should be represented in the
  new runtime.

### Experiment 8: Routing and App Shell

Scope:

- implement `Pages`, `Page`, `NavLink`, route params, query params, and
  programmatic `navigate`;
- support hash routing first, then history routing;
- expose route-derived context variables to compiled expressions.

What we learn:

- how routing state participates in the reactive graph;
- what the minimum provider stack really is;
- how to keep route diagnostics and defended-routing checks compiler-driven.

### Experiment 9: Theming, Layout, and Styling

Scope:

- support core layout properties, theme variables, component parts, theme
  scoping, and enough CSS generation/injection for old visual tests;
- decide which styling work happens at compile time and which stays runtime.

What we learn:

- how to preserve XMLUI's theme contract without copying the old style registry
  wholesale;
- how SSG and hydration requirements affect CSS delivery;
- whether metadata can unify runtime styling, docs, and tooling.

### Experiment 10: Standalone Runtime Compilation

Scope:

- build a standalone bundle that fetches `Main.xmlui` and component files,
  compiles them in the browser, and runs an app without Vite;
- preserve the old static folder shape and extension-script loading model for
  the implemented subset.

What we learn:

- whether the compiler can run acceptably in-browser;
- which compiler pieces must be shared between Node and browser environments;
- how to keep standalone behavior compatible while production builds precompile
  everything.

### Experiment 11: Production Build and Artifact Shape

Scope:

- precompile XMLUI files during build;
- emit optimized JavaScript, CSS, metadata, source maps, and package exports;
- compare startup and interaction performance against old XMLUI and a React
  baseline.

What we learn:

- whether the architecture meets the 50x expression/handler performance goal;
- which artifacts are public compatibility contracts;
- how much runtime code can disappear in production.

### Experiment 12: Static Site Generation and Hydration

Scope:

- prerender static routes to HTML;
- preserve correct first paint before JavaScript loads;
- hydrate with compiled runtime modules;
- keep fallback and `preview-ssg` behavior compatible.

What we learn:

- how the compiler/runtime split supports server-side rendering;
- how CSS, route discovery, search indexes, and prefetched content should be
  represented in build artifacts;
- what hydration constraints must feed back into component/runtime design.

### Experiment 13: Tooling and Metadata Unification

Scope:

- make the compiler IR produce metadata for VS Code completions, diagnostics,
  docs generation, type contracts, and examples;
- rebuild enough of the VS Code extension and docs pipeline to consume the new
  metadata shape.

What we learn:

- whether one metadata source can replace the old parallel systems;
- how diagnostics should be shared between editor, Vite, tests, and runtime;
- how much old docs/language-server infrastructure can be preserved.

### Experiment 14: Extension Packages and Public API Compatibility

Scope:

- support extension packages with components, functions, themes, metadata, and
  standalone/Vite registration;
- port at least one first-party extension and its tests.

What we learn:

- whether the new renderer/component API is stable enough for external authors;
- how namespace and theme-prefix rules fit the compiler architecture;
- which compatibility shims are needed for old extension packages.

### Experiment 15: Full Compatibility Sweep

Scope:

- run the old unit, E2E, integration, docs-example, and package tests against
  the new implementation;
- triage failures into implemented compatibility gaps, intentional deviations,
  and obsolete old tests;
- measure performance, bundle size, startup, and memory.

What we learn:

- whether the rewrite can replace the old framework in real projects;
- which remaining old abstractions were actually necessary;
- where compatibility, performance, and simplicity are still in tension.

## 8. Initial Milestones

1. Implement the first migration experiment in Vite dev mode.
2. Build an inventory of compatibility surfaces from the old docs, tests,
   metadata, package scripts, and generated outputs.
3. Design the shared compiler intermediate representation for markup, script,
   dependencies, metadata, routes, assets, and diagnostics.
4. Prototype compiled expression evaluation and compare it with the old
   interpreter using compatibility fixtures and benchmarks.
5. Prototype compiled event handlers, including async semantics, action lookup,
   state writes, error behavior, and source maps.
6. Design the explicit reactive graph and invalidation scheduler that replaces
   proxy-based change detection.
7. Define the minimum runtime boundary for components, data operations, routing,
   theming, forms, extension registration, diagnostics, and hydration.
8. Recreate the three execution models: standalone, Vite dev, and production.
9. Recreate SSG on top of the compiler/runtime split.
10. Restore tooling compatibility: CLI, create app utility, VS Code extension,
   metadata generation, docs generation, playground, and integration tests.

## 9. Guiding Rule

The rewrite may radically change XMLUI internals, but it must not surprise
existing XMLUI users. The public model remains declarative XMLUI; the new engine
should simply make that model faster, smaller, easier to reason about, and more
consistent across runtime, build, tooling, and documentation.
